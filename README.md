

## 🔹 Завдання 1: Дослідження активних UDP з'єднань

### Відповіді на питання:

* **Які стани ви бачите для UDP сокетів? Чому там немає "LISTENING" або "ESTABLISHED"?**
  Для UDP сокетів у виведенні `netstat` стан взагалі зазвичай залишається пустим (або показує просто `*:*`). Там немає станів `LISTENING` чи `ESTABLISHED`, тому що UDP — це протокол без встановлення з'єднання (connectionless). Він не робить рукостискань і не тримає постійний зв'язок. Сокет просто відкриває порт і готовий приймати чи відправляти окремі пакети (дейтаграми) кому завгодно в будь-який момент.

* **Які сервіси у вашій системі використовують UDP?**
  Зазвичай це системні служби: порт `53` (DNS для перетворення імен сайтів в IP), порт `123` (NTP для синхронізації часу комп'ютера з інтернетом), порти `1900` та `5353` (SSDP та mDNS для пошуку пристроїв у локальній мережі, наприклад, смарт-телевізорів чи принтерів).

* **Чому браузер майже не використовує UDP для звичайного серфінгу (до появи HTTP/3)?**
  Тому що для відображення сайтів (HTML, CSS, картинки) критично важливо, щоб усі дані дійшли на 100% і без помилок. Якщо в UDP загубиться шматок коду сайту, сторінка просто зламається. TCP гарантує доставку та правильний порядок, тому класичні протоколи HTTP/1.1 та HTTP/2 працюють суто на TCP.

## 🔹 Завдання 2: Популярні UDP сервіси

### 1. Таблиця популярних UDP портів (Заповнена)

| Порт | Сервіс | Опис |
| :--- | :--- | :--- |
| **53** | DNS | Domain Name System (Перетворення доменів в IP-адреси) |
| **67, 68** | DHCP | Dynamic Host Configuration Protocol (Автоматична роздача IP) |
| **123** | NTP | Network Time Protocol (Синхронізація часу) |
| **161** | SNMP | Simple Network Management Protocol (Моніторинг мережевих пристроїв) |
| **500** | ISAKMP | VPN / IPsec (Встановлення безпечного з'єднання та обмін ключами) |
| **1900** | SSDP | Simple Service Discovery Protocol (Пошук "розумних" пристроїв у локалці) |
| **443** | HTTP/3 (QUIC) | Сучасний швидкий веб-серфінг на базі UDP |

### 2. Відповіді на питання:

* **Чому DNS використовує переважно UDP? В яких випадках він переходить на TCP?**
  DNS використовує UDP, бо запити та відповіді дуже маленькі (вміщаються в один пакет). Створювати TCP-з'єднання (три кроки рукостискання) заради одного короткого запиту — це занадто довго і створює велике навантаження на сервери. 
  **Перехід на TCP** відбувається у двох випадках: коли розмір відповіді перевищує 512 байт (наприклад, при передачі великих DNSSEC ключів) або під час синхронізації зон між самими DNS-серверами (Zone Transfer).

* **Як працює DHCP, якщо у клієнта ще немає IP-адреси?**
  Він використовує широкомовлення (**Broadcast**). Новий пристрій кричить на всю локальну мережу (надсилає пакет на IP `255.255.255.255`), повідомляючи, що йому потрібна адреса. Цей крик чують всі пристрої, але реагує тільки DHCP-сервер (роутер), який у відповідь виділяє і надсилає клієнту доступний IP.

* **Что таке QUIC і яке відношення він має до UDP?**
  QUIC (Quick UDP Internet Connections) — це новий мережевий протокол від Google, який працює **поверх UDP**. Він бере швидкість UDP (без довгого встановлення зв'язку) і додає до нього надійність від TCP (контроль втрати пакетів та повторну відправку), а також вбудоване шифрування TLS 1.3. Саме на QUIC базується сучасний стандарт **HTTP/3**.

## 🔹 Завдання 3: Експерименти з надсиланням UDP пакетів

### Відповіді на питання:

* **Що станеться, якщо ви надішлете UDP пакет на порт, який ніхто не слухає? Чи отримає відправник якусь помилку?**
  Сам протокол UDP промовчить, бо він не контролює доставку. Проте операційна система отримувача (через протокол ICMP) засіче, що порт закритий, і відправить назад пакет `Port Unreachable`. Через це утиліта відправника (наприклад, `ncat` на Windows) викине помилку в консоль: `Ncat: An existing connection was forcibly closed by the remote host.`

* **Чи зберігається порядок повідомлень, якщо надіслати 10 пакетів поспіль через nc?**
  На локальному комп'ютері (`127.0.0.1`) або в ідеальній домашній локалці вони прийдуть по порядку, бо пакетам немає де блукати. Але в реальному інтернеті — ні, порядок не гарантується. Кожен пакет може летіти своїм маршрутом через різні роутери, тому пакет №5 може прилетіти раніше за пакет №2.

## 🔹 Завдання 7: UDP Broadcast Chat

### Відповіді на питання:

* **Чому TCP не підтримує Broadcast?**
  TCP заснований на концепції індивідуальної сесії між двома конкретними точками (**Unicast**). Перед передачею даних TCP обов'язково робить тристороннє рукостискання (3-Way Handshake), контролює доставку кожного байта, вимагає підтвердження отримання (`ACK`) та підлаштовує швидкість. Робити рукостискання та синхронізувати потік одночасно з усіма комп'ютерами в мережі технічно неможливо.

* **Яка різниця між Broadcast та Multicast?**
  * **Broadcast (Широкомовлення):** Пакет надсилається взагалі **всім** пристроям у поточній локальній мережі (`255.255.255.255`). Кожен комп'ютер у мережі змушений витрачати ресурси, щоб відкрити цей пакет і розібрати його, навіть якщо він йому непотрібен. Це сильно забиває мережу.
  * **Multicast (Багатоадресне мовлення):** Пакет надсилається лише **визначеній групі** пристроїв, які "підписалися" на спеціальну IP-адресу (діапазон `224.0.0.0` – `239.255.255.255`). Роутери самостійно фільтрують трафік і шлють пакет тільки тим пристроям, які входять у цю групу (використовується для IPTV та онлайн-трансляцій).
